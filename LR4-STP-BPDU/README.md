## Chapter II

### Теоретические основы лабораторной работы

#### Что такое петля на канальном уровне

Если несколько коммутаторов образуют цикл, широковещательные кадры могут бесконечно циркулировать по сети. В отличие от IP, на канальном уровне нет поля TTL, поэтому кадр сам по себе не «затухает». Это приводит к следующим проблемам:

- **широковещательный шторм** — exponential рост количества broadcast-кадров в сети;
- **дублирование кадров** — unicast-кадры достигают получателя несколько раз;
- **перегрузка оборудования** — коммутаторы расходуют ресурсы на пересылку бесполезного трафика;
- **нестабильность MAC-таблиц** — коммутаторы постоянно обновляют записи, что затрудняет корректную пересылку.

Именно поэтому в сетях, где существует физическая избыточность связей, необходим механизм предотвращения петель.

#### Назначение STP

Spanning Tree Protocol (`STP`, стандарт IEEE 802.1D) строит логическое дерево без петель поверх физически избыточной топологии. Для этого протокол:

1. выбирает **корневой коммутатор** (`Root Bridge`) на основе наименьшего Bridge ID;
2. рассчитывает **кратчайшие пути** от каждого коммутатора до корневого;
3. **блокирует** избыточные связи, которые могли бы образовать петлю.

При этом логическая топология превращается в дерево: существует ровно один активный путь между любыми двумя узлами. Если один из активных линков отказывает, STP пересчитывает дерево и активирует ранее заблокированный резервный путь.

#### Роль BPDU

BPDU (`Bridge Protocol Data Unit`) — это служебные кадры, которыми коммутаторы обмениваются для расчёта дерева STP. Существует два типа BPDU:

- **Configuration BPDU** — рассылаются корневым коммутатором и содержат информацию о текущей топологии дерева;
- **TCN BPDU** (`Topology Change Notification`) — отправляются некорневыми коммутаторами при обнаружении изменения топологии.

В кадрах Configuration BPDU присутствуют следующие ключевые поля:

| Поле | Назначение |
|---|---|
| `Root ID` | идентификатор корневого коммутатора |
| `Bridge ID` | идентификатор отправителя BPDU |
| `Port ID` | идентификатор порта, с которого отправлен BPDU |
| `Root Path Cost` | стоимость пути до корневого коммутатора |
| `Hello Time` | интервал отправки BPDU |
| `Max Age` | время жизни BPDU |
| `Forward Delay` | задержка перед переходом порта в состояние forwarding |

Именно по BPDU можно увидеть, что STP включён и работает. BPDU отправляются периодически (обычно каждые 2 секунды) и позволяют обнаружить изменения в топологии.

#### Почему резервный линк важен

Физическая избыточность каналов полезна, так как позволяет пережить обрыв одного соединения. STP блокирует резервный путь до аварии, но при отказе основного линка перестраивает дерево и активирует альтернативный маршрут.

Процесс восстановления включает:

1. обнаружение обрыва (отсутствие BPDU от соседа);
2. пересчёт дерева STP;
3. перевод заблокированного порта в состояние listening → learning → forwarding;
4. восстановление связности.

Именно этот процесс предстоит наблюдать в ходе лабораторной работы.

#### Что необходимо понять по итогам теории

После изучения теоретического материала студент должен понимать:

1. почему петля на L2 опасна;
2. какую задачу решает STP;

5. что происходит с резервным линком при работе STP;
6. почему после отказа одного линка сеть может продолжить работу.

### Общие сведения о GNS3 и Open vSwitch

#### Среда GNS3

В работе используется кольцевая топология из трёх Linux-узлов, на которых установлен Open vSwitch, и двух конечных узлов. Такой подход позволяет воспроизвести STP без необходимости загружать отдельные образы физических коммутаторов.

Linux-узлы `SW1`, `SW2`, `SW3` должны иметь по три сетевых адаптера каждый. Узлы `PC1`, `PC2` выполняются на базе VPCS.

#### Команды VPCS, необходимые для проверки связности

Для настройки конечных узлов и проверки работы дерева STP используются стандартные команды VPCS:

| Команда | Назначение |
|---|---|
| `ip <адрес>/<маска>` | назначить IP-адрес и маску |
| `show ip` | показать текущие IP-настройки |
| `ping <IP-адрес>` | выполнить проверку связности |
| `ping <IP-адрес> -c <N>` | отправить серию из `N` ICMP-запросов |
| `save` | сохранить конфигурацию |

#### Настройка Open vSwitch для STP

На всех коммутирующих узлах необходимо:

1. установить `openvswitch-switch`;
2. создать bridge `br0`;
3. добавить в него интерфейсы `eth0`, `eth1`, `eth2`;
4. включить STP.

Основные команды OVS, необходимые для данной работы:

| Команда | Назначение |
|---|---|
| `apt update && apt install -y openvswitch-switch` | установка OVS |
| `ovs-vsctl add-br br0` | создание bridge |
| `ovs-vsctl add-port br0 eth0` | добавление порта в bridge |
| `ovs-vsctl set bridge br0 stp_enable=true` | включение STP на bridge |
| `ovs-vsctl show` | отображение текущей конфигурации |
| `ovs-ofctl show br0` | отображение состояния портов |

#### Действия в GNS3, необходимые для моделирования отказа линка

В данной лабораторной работе недостаточно только настроить коммутаторы: студенту также нужно уметь выполнить несколько базовых действий в интерфейсе GNS3.

| Действие | Назначение |
|---|---|
| запуск всех узлов (`Start/Resume`) | перевести стенд в рабочее состояние |
| открытие консоли узла (`Console`) | ввести команды VPCS или Linux |
| запуск захвата на выбранном линке (`Start Capture`) | получить pcap с BPDU |
| остановка захвата (`Stop Capture`) | завершить и сохранить наблюдение |
| временное отключение линка через контекстное меню соединения | смоделировать отказ физического канала |
| повторное включение линка или возврат соединения | убедиться в восстановлении топологии |

### Задание 1. Сборка кольцевой топологии и настройка STP

Для этого задания используется параметр `A`, который вычисляется по формуле:

- `A = ((N - 1) mod 4) + 1`

Вариант адресации по параметру `A`:

| `A` | Подсеть | `PC1` | `PC2` |
|---|---|---|---|
| 1 | `192.168.40.0/24` | `192.168.40.10/24` | `192.168.40.20/24` |
| 2 | `192.168.50.0/24` | `192.168.50.10/24` | `192.168.50.20/24` |
| 3 | `10.10.10.0/24` | `10.10.10.10/24` | `10.10.10.20/24` |
| 4 | `172.16.1.0/24` | `172.16.1.10/24` | `172.16.1.20/24` |

Используются следующие узлы:

- `SW1`, `SW2`, `SW3` — Linux-узлы с Open vSwitch (по 3 сетевых адаптера);
- `PC1`, `PC2` — VPCS.

Топология:

Кольцо коммутаторов:

1. `SW1(eth0) — SW2(eth0)`
2. `SW2(eth1) — SW3(eth0)`
3. `SW3(eth1) — SW1(eth1)`

Подключение конечных узлов:

1. `PC1 — SW1(eth2)`
2. `PC2 — SW3(eth2)`

Назначить IP-адреса `PC1` и `PC2` в соответствии с параметром `A`.

Порядок настройки OVS на `SW1`, `SW2`, `SW3`:

1. установить `openvswitch-switch`;
2. создать bridge `br0`;
3. добавить все три интерфейса (`eth0`, `eth1`, `eth2`) в `br0`;
4. включить STP: `ovs-vsctl set bridge br0 stp_enable=true`;
5. сохранить конфигурацию.

В отчёт по данному заданию включить:

- схему кольцевой топологии;
- таблицу адресации;
- команды настройки OVS.

В составе сдаваемых материалов должны быть представлены:

- проект GNS3;
- текстовые команды настройки OVS.
