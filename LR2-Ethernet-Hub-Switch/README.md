## Chapter II

### Теоретические основы лабораторной работы

#### Ethernet II как базовая технология локальной сети

В локальных сетях передача данных на канальном уровне чаще всего осуществляется в формате Ethernet II. Кадр Ethernet II содержит несколько ключевых полей:

| Поле | Назначение |
|---|---|
| `Destination MAC` | MAC-адрес назначения |
| `Source MAC` | MAC-адрес источника |
| `EtherType` | тип вложенного протокола (`0x0806` — ARP, `0x0800` — IPv4) |
| `Payload` | полезная нагрузка |
| `FCS` | контрольная последовательность кадра |

Поле `EtherType` показывает, какой протокол вложен в кадр. Например, значение `0x0806` соответствует ARP, а значение `0x0800` соответствует IPv4. Именно Ethernet-кадр является основным «контейнером» для передачи IP-пакетов в локальной сети.

#### MAC-адреса и типы доставки на L2

На канальном уровне адресация строится на MAC-адресах. MAC-адрес является адресом сетевого интерфейса на канальном уровне и используется для доставки кадров в пределах локального сегмента сети. В Ethernet обычно применяется 48-битный MAC-адрес, записываемый в шестнадцатеричном виде.

В Ethernet используются три базовых типа доставки:

1. **unicast** — кадр направлен одному конкретному получателю;
2. **broadcast** — кадр направлен всем узлам локального сегмента;
3. **multicast** — кадр направлен группе узлов.

Для Ethernet широковещательный MAC-адрес имеет вид `ff:ff:ff:ff:ff:ff`.

#### Протокол ARP и связь с broadcast

Протокол ARP (`Address Resolution Protocol`) используется для определения MAC-адреса по известному IP-адресу в пределах одной локальной сети. Если хост знает IP-адрес соседа, но не знает его MAC-адрес, он формирует ARP Request.

Поскольку MAC-адрес получателя ещё неизвестен, первый ARP-запрос отправляется как broadcast-кадр. Узел, которому принадлежит целевой IP-адрес, отвечает ARP Reply уже в формате unicast.

Таким образом, ARP обеспечивает связь между логической IP-адресацией и фактической передачей кадров Ethernet. Обмен ARP выглядит как короткий предварительный этап перед полезным обменом данными.

#### Hub и Switch: принципиальное различие

Hub работает как повторитель сигнала. Он не анализирует MAC-адреса и просто дублирует пришедший сигнал на все остальные порты. В результате весь трафик сегмента виден на каждом порту, включая порты, к которым подключены узлы, не являющиеся адресатами кадров.

Switch работает на канальном уровне и анализирует MAC-адрес назначения. Он обучается, формируя таблицу соответствий `MAC → порт`, и затем пересылает unicast-кадры только на нужный порт. Broadcast-кадры при этом по-прежнему распространяются по всем активным портам.

Именно поэтому на контрольной точке наблюдения картина трафика будет разной:

- в стенде с Hub на линке `PC2 - Hub` будет виден весь трафик, включая чужой unicast `PC1 ↔ PC3`;
- в стенде со Switch на линке `PC2 - Switch` чужой unicast `PC1 ↔ PC3` будет отсутствовать, а broadcast-кадры (например, ARP Request) будут по-прежнему видны.

Это ключевое различие и предстоит подтвердить в ходе лабораторной работы.

#### Что необходимо понять по итогам теории

После изучения теоретического материала студент должен понимать:

1. из каких полей состоит Ethernet-кадр;
2. чем MAC-адрес отличается от IP-адреса;
3. почему ARP Request является широковещательным;
4. почему ARP Reply является одноадресным;
5. почему в стенде с Hub на контрольном порту виден чужой unicast-трафик;
6. почему в стенде со Switch на контрольном порту не виден чужой unicast-трафик.

### Общие сведения о GNS3, VPCS, Hub и Ethernet Switch

#### GNS3

В данной лабораторной работе GNS3 используется для построения двух минимальных топологий:

1. `PC1 — Hub — PC2 — PC3` (стенд A);
2. `PC1 — Switch — PC2 — PC3` (стенд B).

Все конечные узлы выполняются на базе VPCS. В первом стенде используется встроенный Hub GNS3, во втором — встроенный Ethernet Switch.

#### VPCS

VPCS (`Virtual PC Simulator`) представляет собой лёгкий виртуальный ПК, предназначенный для учебных сетевых стендов. Основные команды VPCS, необходимые для данной работы:

| Команда | Назначение |
|---|---|
| `ip <адрес>/<маска>` | назначить IP-адрес и маску |
| `show ip` | показать текущие IP-настройки |
| `ping <IP-адрес>` | проверить связность с указанным узлом |
| `ping <IP-адрес> -c <N>` | отправить `N` ICMP-запросов |
| `arp` | вывести ARP-таблицу |
| `clear arp` | очистить ARP-таблицу |
| `save` | сохранить конфигурацию |
| `?` или `help` | вывести список доступных команд |

#### Hub в GNS3

`Ethernet Hub` в GNS3 представляет собой встроенный виртуальный повторитель. Он не анализирует MAC-адреса и просто пересылает все кадры, полученные на один порт, на все остальные порты. В данной лабораторной работе Hub используется для демонстрации поведения общей среды передачи.

#### Ethernet Switch в GNS3

`Ethernet Switch` в GNS3 представляет собой встроенный виртуальный коммутатор канального уровня. Он автоматически обучается MAC-адресам и пересылает unicast-кадры только на нужный порт. Специальная настройка коммутатора для данной работы не требуется: достаточно добавить его в проект и соединить с узлами VPCS.

### Общие сведения о Wireshark

Wireshark используется для сравнительного анализа двух pcap-файлов. При анализе необходимо уметь:

1. фильтровать пакеты по типу протокола;
2. находить broadcast-кадры по MAC-адресу назначения;
3. находить кадры обмена между конкретными узлами;
4. фиксировать номера кадров и выписывать значения ключевых полей.

При работе с Wireshark необходимо различать три основные панели:

1. список пакетов;
2. дерево полей выбранного пакета;
3. байтовое представление пакета.

Полезные фильтры отображения для данной лабораторной работы:

| Фильтр | Назначение |
|---|---|
| `arp` | показать только ARP-кадры |
| `icmp` | показать только ICMP-пакеты |
| `eth.dst == ff:ff:ff:ff:ff:ff` | показать широковещательные кадры |
| `ip.addr == 192.168.20.11` | отфильтровать трафик по IP-адресу PC1 |
| `ip.addr == 192.168.20.33` | отфильтровать трафик по IP-адресу PC3 |

### Задание 4. Сравнительный анализ Hub и Switch

Для этого задания используется параметр `D`, который вычисляется по формуле:

- `D = ((N - 1) mod 4) + 1`

Вариант вопроса для сравнительного анализа по параметру `D`:

- `D = 1` - почему на контрольном линке в стенде с Hub виден unicast-трафик, не адресованный наблюдателю, а со Switch - нет.
- `D = 2` - как коммутатор определяет, на какой порт переслать unicast-кадр, и к чему это приводит на практике.
- `D = 3` - почему `ARP Request` виден на всех портах обоих стендов, несмотря на различие Hub/Switch.
- `D = 4` - как изменится наблюдаемая картина, если перед захватом выполнить `ping` со всех узлов ко всем и заполнить MAC-таблицу коммутатора.

В отчёте необходимо письменно ответить на следующие общие пункты:

1. Какие кадры обмена `PC1 ↔ PC3` видны на контрольном линке в стенде с Hub?
2. Какие кадры обмена `PC1 ↔ PC3` видны на контрольном линке в стенде со Switch?
3. Почему broadcast-кадры видны в обоих стендах?
4. Почему unicast-трафик `PC1 ↔ PC3` виден только в стенде с Hub?

Дополнительно выполнить аналитический пункт по параметру `D`.

В отчёт по данному заданию включить:

- сравнительный анализ с отсылками к конкретным номерам кадров из pcap;
- выполнение аналитического пункта по параметру `D`;
- итоговый вывод о различии между Hub и Switch.
